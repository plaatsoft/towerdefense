284 /*
285  * Y2K compliance in libpng:
286  * =========================
287  *
288  * May 6, 2010
289  *
290  * Since the PNG Development group is an ad-hoc body, we can't make
291  * an official declaration.
292  *
293  * This is your unofficial assurance that libpng from version 0.71 and
294  * upward through 1.4.2 are Y2K compliant. It is my belief that earlier
295  * versions were also Y2K compliant.
296  *
297  * Libpng only has three year fields. One is a 2-byte unsigned integer
298  * that will hold years up to 65535. The other two hold the date in text
299  * format, and will hold years up to 9999.
300  *
301  * The integer is
302  * "png_uint_16 year" in png_time_struct.
303  *
304  * The strings are
305  * "png_charp time_buffer" in png_struct and
306  * "near_time_buffer", which is a local character string in png.c.
307  *
308  * There are seven time-related functions:
309  * png.c: png_convert_to_rfc_1123() in png.c
310  * (formerly png_convert_to_rfc_1152() in error)
311  * png_convert_from_struct_tm() in pngwrite.c, called in pngwrite.c
312  * png_convert_from_time_t() in pngwrite.c
313  * png_get_tIME() in pngget.c
314  * png_handle_tIME() in pngrutil.c, called in pngread.c
315  * png_set_tIME() in pngset.c
316  * png_write_tIME() in pngwutil.c, called in pngwrite.c
317  *
318  * All handle dates properly in a Y2K environment. The
319  * png_convert_from_time_t() function calls gmtime() to convert from system
320  * clock time, which returns (year - 1900), which we properly convert to
321  * the full 4-digit year. There is a possibility that applications using
322  * libpng are not passing 4-digit years into the png_convert_to_rfc_1123()
323  * function, or that they are incorrectly passing only a 2-digit year
324  * instead of "year - 1900" into the png_convert_from_struct_tm() function,
325  * but this is not under our control. The libpng documentation has always
326  * stated that it works with 4-digit years, and the APIs have been
327  * documented as such.
328  *
329  * The tIME chunk itself is also Y2K compliant. It uses a 2-byte unsigned
330  * integer to hold the year, and can hold years as large as 65535.
331  *
332  * zlib, upon which libpng depends, is also Y2K compliant. It contains
333  * no date-related code.
334  *
335  * Glenn Randers-Pehrson
336  * libpng maintainer
337  * PNG Development Group
338  */
